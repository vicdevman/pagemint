Pagemint: Full Product Requirements Document (PRD)

On-Chain Website-as-a-Service, Discovery, and Intelligent Engagement Protocol on Monad

1. Executive Summary

Short Description (The Hook)

pagemint is a decentralized Website-as-a-Service platform on Monad that instantly transforms any smart contract address into a highly premium, interactive, and token-gated landing page. By shifting page configurations, monetization mechanics, and voting credits directly on-chain, pagemint eliminates Web2 setup friction and turns passive token addresses into active community portals.

Long Description (The Vision)

In the high-velocity Web3 ecosystem—especially within emerging networks like Monad—speed to market and cryptographic credibility are everything. Standard Web2 landing pages are slow to build, expensive to host, and fundamentally disconnected from the on-chain states they represent. Their metrics (like social followers or total value locked) are easily manipulated, and building gated access for community members requires complex API middleware.

pagemint solves this by establishing an on-chain registry that turns smart contract addresses into dynamic, editable, and highly functional landing pages. Users "mint" their page by binding a target contract address to our master registry on the Monad Testnet.

The system operates via a collaborative dual-agent architecture:

Agent Alpha (The Contract Parser): Analyzes target contract structures, parses functional ABIs, and automatically identifies features (e.g., Minting, Staking, Swapping) to dynamically construct Web3 transaction widgets.

Agent Beta (The Dynamic Designer): Renders hyper-modern, high-converting frontends using Shadcn-style components, avoiding generic AI templates.

To make the platform self-sustaining and native to Web3, pagemint implements an On-Chain Credit System powered by platform fees and Trust Staking. When users interact with pages or stake native MON to support projects, they earn on-chain credits used to execute deep contract analysis, page customizability, and LLM-powered engine operations.

Agent Alpha: The Contract Parser (Build & Analysis Phase)

Agent Alpha operates as the developer's technical translator. Its goal is to dissect any contract address and make its capabilities understandable to the frontend.

ABI & Pattern Analysis: Alpha reads the target contract interface. It searches for standard signatures (ERC-20, ERC-721, ERC-1155) as well as custom execution structures (e.g., functions containing phrases like stake, deposit, burn, mint, swap, lend).

Classification Logic:

Token Asset: If standard ERC-20 functions are discovered, Alpha flags this as an asset and passes swap/buy parameters to Agent Beta.

Protocol Asset: If the contract exposes state variables like totalAssets, apy, paused or write methods like deposit(), Alpha flags this as a protocol and creates a custom interaction schema.

Dynamic Feature Mapping: Instead of rendering static text, Alpha outputs a JSON Schema mapping the available interactive endpoints of the target contract.

Example: If it finds a stake(uint256 amount) function, it generates a data structure instructing Agent Beta to render an active staking input form.

Agent Beta: The Dynamic Designer (UX & UI Render Engine)

Agent Beta is responsible for translating the data schemas generated by Agent Alpha into a highly aesthetic, premium Web3 landing page.

Aesthetic Execution: Using modern Tailwind utility design tokens and a Shadcn/Radix-style design scheme, Beta ensures the generated page feels custom-tailored, premium, and distinct.

Interactive Action Widgets:

The Smart Swap/Buy Module: For standard token pages, Beta renders a direct buy/swap widget allowing users to swap native MON for the target token.

The Direct Protocol Playground: For protocol pages, Beta renders direct interaction modules (e.g., input forms to deposit funds directly into the parsed vault or stake in the pool) without sending the user away to a separate app.

Decoupled Gating Layouts: Beta reads the gating configurations defined by the creator on-chain. If the gating condition is active, premium documents (e.g., internal roadmap, audit logs, pitch decks) are blurred until a balance check of the specified gateToken returns successful.

3. Core Monetization & Mechanics

To move away from standard database solutions, pagemint integrates value-exchange mechanisms directly into the Solidity smart contract.

A. The Minting Platform Fee

To mint a page, creators must pay a native MINT_FEE in MON (e.g., $0.01$ MON).

This fee goes directly to the protocol treasury to bootstrap the platform's execution layer and fund the API key infrastructure needed for real-time code parsing.

B. On-Chain Credit Accounting System

To run complex parser executions (which utilize Agent Alpha's LLM components), the system uses an on-chain credit registry (userCredits).

Initial Boost: Upon connecting a wallet for the first time, users are assigned a small welcome credit balance to test the tool.

Earned Credits: When a developer's page gets high engagement or upvotes, they are rewarded with on-chain credits directly to their address.

Purchasing Credits: Users can burn native MON directly through the contract to buy credits (buyCredits()), which directly funds LLM compute costs on-chain.

C. Trust Staking (The Curation & Voting Engine)

Instead of free, bot-manipulated Web2 "likes," pagemint uses a native micro-staking model:

To upvote or support a minted page, a supporter sends a small amount of native MON (the STAKE_AMOUNT, e.g., $0.05$ MON) along with their vote.

The Incentivized Split:

$80\%$ of the staked MON goes directly to the Page Creator. This incentivizes developers to build highly engaging, accurate, and functional projects.

$20\%$ goes to the Protocol Treasury as recurring platform revenue.

The page creator also receives platform credits for every trust vote cast on their project.

4. Technical Implementation: The Smart Contract

The core state machine must be deployed on the Monad Testnet. This upgraded version decouples the targetContract (the token or protocol contract address) from the gateToken (the token address used for balance checks), allowing creators of non-token protocols to utilize standard ERC-20 assets (such as native MON wrappers or governance tokens) for their gating requirements.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/**
 * @dev Interface for interacting with target ERC20 assets for token gating.
 */
interface IERC20 {
    function balanceOf(address account) external view returns (uint256);
}

contract MonadPageMinter {
    
    struct Page {
        address creator;
        address targetContract;   // The contract representing the project (token, vault, or protocol)
        address gateToken;        // The specific ERC-20 token checked for gating access
        uint256 minTokenToUnlock;// Minimum token balance of gateToken needed to view gated content
        uint256 totalTrustVotes; // Aggregated number of unique trust votes
        uint256 totalStakedValue;// Total MON staked to this page by supporters
        string ipfsMetadataHash; // Hash containing custom layouts, roadmaps, and configurations
        uint256 mintedAt;
        bool isMinted;
    }

    // Protocol configurations
    address public owner;
    uint256 public mintFee = 0.01 ether;       // Fee in MON to mint a page
    uint256 public voteStakeAmount = 0.05 ether; // Amount of MON required to trust-vote
    uint256 public creditCostPerMon = 1000;    // Credits granted per 1 MON purchased

    // Global accounting metrics
    uint256 public totalPagesMinted;
    uint256 public totalProtocolRevenue;

    // Registries
    mapping(address => Page) public registry;                       // targetContract => Page details
    mapping(address => mapping(address => bool)) public hasVoted;    // user => targetContract => voted status
    mapping(address => uint256) public userCredits;                 // user => on-chain computational credits

    // Events
    event PageMinted(
        address indexed targetContract, 
        address indexed creator, 
        address gateToken, 
        uint256 minTokenToUnlock, 
        string ipfsHash
    );
    event PageUpdated(address indexed targetContract, address gateToken, uint256 minTokenToUnlock, string newIpfsHash);
    event TrustVoted(address indexed targetContract, address indexed voter, uint256 currentVotes, uint256 creatorPayout);
    event CreditsPurchased(address indexed user, uint256 amountSpent, uint256 creditsAcquired);
    event CreditsSpent(address indexed user, uint256 amountSpent, string purpose);

    modifier onlyOwner() {
        require(msg.sender == owner, "Only owner can execute");
        _;
    }

    constructor() {
        owner = msg.sender;
    }

    /**
     * @notice Allows users to buy computation/LLM credits directly on-chain using native MON.
     */
    function buyCredits() external payable {
        require(msg.value > 0, "Must send MON to buy credits");
        uint256 creditsEarned = (msg.value * creditCostPerMon) / 1 ether;
        userCredits[msg.sender] += creditsEarned;
        totalProtocolRevenue += msg.value;

        emit CreditsPurchased(msg.sender, msg.value, creditsEarned);
    }

    /**
     * @notice Deducts credits from a user when they trigger Agent Alpha analysis.
     */
    function spendCredits(address _user, uint256 _amount, string calldata _purpose) external {
        require(msg.sender == owner || msg.sender == _user, "Unauthorized spending");
        require(userCredits[_user] >= _amount, "Insufficient credit balance");
        
        userCredits[_user] -= _amount;
        emit CreditsSpent(_user, _amount, _purpose);
    }

    /**
     * @notice Mints a new interactive landing page for a target token/protocol contract.
     * @param _targetContract The contract address representing the project (token or protocol).
     * @param _gateToken The specific ERC-20 token contract to check balance against for gating.
     * @param _minTokenToUnlock Threshold requirement for dynamic content unlocking.
     * @param _ipfsMetadataHash IPFS pointer containing roadmap, styles, and custom configurations.
     */
    function mintPage(
        address _targetContract,
        address _gateToken,
        uint256 _minTokenToUnlock,
        string calldata _ipfsMetadataHash
    ) external payable {
        require(_targetContract != address(0), "Invalid target address");
        require(_gateToken != address(0), "Invalid gate token address");
        require(!registry[_targetContract].isMinted, "Page already minted");
        require(msg.value >= mintFee, "Incorrect mint fee sent");

        registry[_targetContract] = Page({
            creator: msg.sender,
            targetContract: _targetContract,
            gateToken: _gateToken,
            minTokenToUnlock: _minTokenToUnlock,
            totalTrustVotes: 0,
            totalStakedValue: 0,
            ipfsMetadataHash: _ipfsMetadataHash,
            mintedAt: block.timestamp,
            isMinted: true
        });

        totalPagesMinted++;
        totalProtocolRevenue += msg.value;

        // Reward creator with starter credits upon successful registration
        userCredits[msg.sender] += 50; 

        emit PageMinted(_targetContract, msg.sender, _gateToken, _minTokenToUnlock, _ipfsMetadataHash);
    }

    /**
     * @notice Allows the verified creator to edit and update their page configurations.
     */
    function updatePageConfig(
        address _targetContract,
        address _gateToken,
        uint256 _minTokenToUnlock,
        string calldata _newIpfsHash
    ) external {
        require(registry[_targetContract].isMinted, "Page not found");
        require(registry[_targetContract].creator == msg.sender, "Only creator can edit");
        require(_gateToken != address(0), "Invalid gate token address");

        registry[_targetContract].gateToken = _gateToken;
        registry[_targetContract].minTokenToUnlock = _minTokenToUnlock;
        registry[_targetContract].ipfsMetadataHash = _newIpfsHash;

        emit PageUpdated(_targetContract, _gateToken, _minTokenToUnlock, _newIpfsHash);
    }

    /**
     * @notice Trust Staking Vote: Supporter stakes MON to upvote. Splits payout (80% to creator, 20% to protocol).
     */
    function submitTrustVote(address _targetContract) external payable {
        require(registry[_targetContract].isMinted, "Page not found");
        require(!hasVoted[msg.sender][_targetContract], "Already voted");
        require(msg.value >= voteStakeAmount, "Insufficient stake sent");

        Page storage page = registry[_targetContract];
        hasVoted[msg.sender][_targetContract] = true;
        
        page.totalTrustVotes += 1;
        page.totalStakedValue += msg.value;

        // Perform split payouts
        uint256 creatorShare = (msg.value * 80) / 100;
        uint256 protocolShare = msg.value - creatorShare;

        totalProtocolRevenue += protocolShare;
        
        // Transfer 80% directly to project creator
        (bool successCreator, ) = payable(page.creator).call{value: creatorShare}("");
        require(successCreator, "Creator payment failed");

        // Reward both voter and creator with on-chain platform credits
        userCredits[page.creator] += 100; // Reward creator for active engagement
        userCredits[msg.sender] += 10;   // Small rebate for active curation

        emit TrustVoted(_targetContract, msg.sender, page.totalTrustVotes, creatorShare);
    }

    /**
     * @notice Core Access Gate verification logic. 
     * @dev Checks if the viewer has the minimum required amount of the designated gateToken.
     */
    function checkAccess(address _viewer, address _targetContract) external view returns (bool) {
        if (!registry[_targetContract].isMinted) return false;
        Page memory page = registry[_targetContract];
        
        // Query the balance of the designated gateToken instead of the target contract itself
        try IERC20(page.gateToken).balanceOf(_viewer) returns (uint256 balance) {
            return balance >= page.minTokenToUnlock;
        } catch {
            // Fallback: If gateToken isn't a standard ERC20, allow viewing to prevent permanent locks
            return true;
        }
    }

    // Protocol administrative updates
    function withdrawFees() external onlyOwner {
        uint256 balance = address(this).balance;
        require(balance > 0, "No funds to withdraw");
        (bool success, ) = payable(owner).call{value: balance}("");
        require(success, "Withdrawal failed");
    }

    function updateMintFee(uint256 _newFee) external onlyOwner {
        mintFee = _newFee;
    }
}


5. Phased Execution Roadmap

To deliver a polished submission before the deadline, the development plan is organized into progressive phases. Phase 1 is the primary focus for tonight.

🚨 Phase 1: The Core Submission (The 12-Hour Sprint)

Focus entirely on delivering a rock-solid, fully functioning implementation of the primary loop.

Contract Deployment:

Deploy the upgraded MonadPageMinter contract on Monad Testnet.

Save and verify the contract address on the Monad block explorer.

The Generation Flow (Dashboard):

Create a clean dashboard page where users can input:

Target Contract Address (the token or protocol address).

Gate Token Address (the ERC-20 token used to unlock sections).

Unlock Threshold (minimum tokens needed to decrypt/unveil sections).

Custom Project Title, Pitch Description, Logo URL, and social links.

Implement the "Mint Landing Page" button to trigger the mintPage() contract call on Monad Testnet.

The Generated View Flow (Dynamic Page):

Build a dynamic page route /page/[contractAddress].

Live Protocol/Token Metrics: If it is a token, display name, symbol, and total supply. If it is a protocol, render the address, deployment date, and configuration parameters.

Dynamic Swap/Action Widget:

Token View: A quick swap panel allowing users to swap native MON for the target token.

Protocol View: Displays direct, interactive function call buttons matching the protocol's methods.

Gated Content Wrapper: A visual roadmap/deck container checking balance using checkAccess(). If access is false, blur the contents with the prompt: "Locked: Requires $[X]$ $[GateToken]$ to unlock."

Trust Voting Button: Add a button calling submitTrustVote(), which executes the split payout to the creator and updates the global metrics.

Leaderboard Directory:

Create a master directory page sorting all minted pages by totalTrustVotes fetched directly from the registry contract.

🚀 Phase 2: Post-Hackathon Integration

Agent Alpha Automated Parser Engine:

Integrate an automated backend parser framework to read target contract bytecodes, parse ABI interfaces, and map interactive write endpoints directly to UI components.

On-Chain Credit Burning:

Integrate the dynamic credit accounting system where advanced customization and deep security analyses deduct credits from the user's registry state.
